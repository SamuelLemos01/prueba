Tipos de repositorios

Un repositorio, o repo, es un tipo de almacenamiento digital centralizado que los desarrolladores utilizan para realizar y administrar cambios en el código fuente de una aplicación. Los desarrolladores tienen que almacenar y compartir carpetas, archivos de texto y otros tipos de documentos al desarrollar software. Un repositorio cuenta con características que permiten a los desarrolladores rastrear con facilidad cambios en el código, editar archivos de manera simultánea y colaborar de forma eficiente en el mismo proyecto desde cualquier ubicación. Existen dos tipos de repositorio, estos son:

Repositorio Web

Son estructuras web que permiten almacenar, organizar, preservar y difundir la producción intelectual de una institución. Los repositorios institucionales se enfocan en la producción académica de una institución, mientras que los repositorios temáticos se enfocan en un campo disciplinar o asunto específico.

Repositorio Local

Son contenedores de archivos que se encuentran en una máquina local y son utilizados por un usuario individual. En un repositorio local se pueden guardar proyectos con diferentes archivos, hacer modificaciones y crear versiones.

Diferencias entre Repositorios Web y Local

  1. Accesibilidad: El repositorio web es accesible desde cualquier lugar con internet, mientras el local solo desde tu computadora.
  2. Colaboración: El web permite trabajo en equipo simultáneo, el local es principalmente para trabajo individual.
  3. Respaldo: El web tiene copias de seguridad automáticas en servidores, el local depende de tus propias copias.
  4. Conectividad: El web requiere conexión a internet para funcionar, el local trabaja sin conexión.
  5. Espacio: El web suele tener límites de almacenamiento según el plan, el local solo depende de tu disco duro.
  6. Velocidad: El local es más rápido al no depender de la velocidad de internet.
  7. Control: El local te da control total sobre tu código, el web depende de políticas del servicio.
  8. Costos: El web puede tener costos de suscripción, el local es generalmente gratuito.

Que es Git y GitHub

Git

Git es un sistema de control de versiones (SCV) de código abierto que permite hacer seguimiento de los cambios en el código fuente de un proyecto. Es una herramienta para getionar proyectos y que permite a varios desarrolladores trabajar en conjunto en el desarrollo de software y aplicaciones.

Sitio Oficial
GitHub

GitHub es una plataforma donde puedes almacenar, compartir y trabajar junto con otros usuarios para escribir código. Almacenar tu código en un "repositorio" en GitHub te permite: Presentar o compartir el trabajo. Seguir y administrar los cambios en el código a lo largo del tiempo.

Sitio Oficial

Diferencias entre Git y GitHub

  1. Naturaleza: Git es un sistema de control de versiones, GitHub es una plataforma web que usa Git.
  2. Instalación: Git se instala localmente en tu computadora, GitHub es un servicio web al que accedes por navegador.
  3. Funcionamiento: Git maneja el versionamiento del código, GitHub proporciona almacenamiento y funciones sociales.
  4. Conexión: Git funciona sin internet, GitHub requiere conexión web constante.
  5. Propiedad: Git es software libre creado por Linus Torvalds, GitHub es una empresa propiedad de Microsoft.
  6. Funciones sociales: Git no tiene características sociales, GitHub permite seguir proyectos y desarrolladores.
  7. Interfaz: Git se usa principalmente por línea de comandos, GitHub ofrece interfaz gráfica web.
  8. Colaboración: Git permite colaboración básica, GitHub añade funciones como issues, pull requests y wikis.

Ramas de Git

Una rama en Git es una línea independiente de desarrollo que permite trabajar en diferentes versiones del código simultáneamente sin afectar la rama principal. Es como una copia del código que puedes modificar de forma segura. Tipos de ramas mas comunes en Git:

  • Es la rama principal por defecto
  • Contiene el código estable y productivo

  • Rama de desarrollo principal
  • Donde se integran las nuevas características antes de pasar a main

  • Ramas temporales para desarrollar nuevas funcionalidades
  • Ejemplo: feature/login, feature/payment-system

  • Para corregir errores urgentes en producción
  • Se crean desde main y se fusionan tanto en main como en develop

  • Para preparar una nueva versión del software
  • Permite últimos ajustes antes de pasar a producción

Comandos basicos de Git

  1. git add: Mueve los cambios del directorio de trabajo al área del entorno de ensayo. Así puedes preparar una instantánea antes de confirmar en el historial oficial.
  2. rama de git: Este comando es tu herramienta de administración de ramas de uso general. Permite crear entornos de desarrollo aislados en un solo repositorio.
  3. Git Checkout: Además de extraer las confirmaciones y las revisiones de archivos antiguas, git checkout también sirve para navegar por las ramas existentes. Combinado con los comandos básicos de Git, es una forma de trabajar en una línea de desarrollo concreta.
  4. git clean: Elimina los archivos sin seguimiento de tu directorio de trabajo. Es la contraparte lógica de git reset, que normalmente solo funciona en archivos con seguimiento.
  5. git clone: Crea una copia de un repositorio de Git existente. La clonación es la forma más habitual de que los desarrolladores obtengan una copia de trabajo de un repositorio central.
  6. git commit: Confirma la instantánea preparada en el historial del proyecto. En combinación con git add, define el flujo de trabajo básico de todos los usuarios de Git.
  7. git init: Inicializa un nuevo repositorio de Git. Si quieres poner un proyecto bajo un control de revisiones, este es el primer comando que debes aprender.
  8. git log: Permite explorar las revisiones anteriores de un proyecto. Proporciona varias opciones de formato para mostrar las instantáneas confirmadas.
  9. git pull: Este comando es la versión automatizada de git fetch. Descarga una rama de un repositorio remoto e inmediatamente la fusiona en la rama actual. Este es el equivalente en Git de svn update.
  10. git pushEnviar (push) es lo opuesto a recuperar (fetch), con algunas salvedades. Permite mover una o varias ramas a otro repositorio, lo que es una buena forma de publicar contribuciones. Es como svn commit, pero envía una serie de confirmaciones en lugar de un solo conjunto de cambios.
  11. git reset:Deshace los cambios en los archivos del directorio de trabajo. El restablecimiento permite limpiar o eliminar por completo los cambios que no se han enviado a un repositorio público.
  12. git revert: Permite deshacer una instantánea confirmada. Si descubres una confirmación errónea, revertirla es una forma fácil y segura de eliminarla por completo del código base.
  13. git statusMuestra el estado del directorio en el que estás trabajando y la instantánea preparada. Lo mejor es que lo ejecutes junto con git add y git commit para ver exactamente qué se va a incluir en la próxima instantánea.
  14. git remoteEs un comando útil para administrar conexiones remotas. En lugar de pasar la URL completa a los comandos fetch, pull y push, permite usar un atajo más significativo.
  15. git rebase: Un cambio de base con git rebase permite mover las ramas, lo que ayuda a evitar confirmaciones de fusión innecesarias. El historial lineal resultante suele ser mucho más fácil de entender y explorar.
  16. git rebase -i: La marca -i se usa para iniciar una sesión de cambio de base interactivo. Esto ofrece todas las ventajas de un cambio de base normal, pero te da la oportunidad de añadir, editar o eliminar confirmaciones sobre la marcha.
  17. Git merge: Es una forma eficaz de integrar los cambios de ramas divergentes. Después de bifurcar el historial del proyecto con git branch, git merge permite unirlo de nuevo.
  18. git fetch:Con este comando, se descarga una rama de otro repositorio junto con todas sus confirmaciones y archivos asociados. Sin embargo, no intenta integrar nada en el repositorio local. Esto te permite inspeccionar los cambios antes de fusionarlos en tu proyecto.
  19. git config: Este comando va bien para establecer las opciones de configuración para instalar Git. Normalmente, solo es necesario usarlo inmediatamente después de instalar Git en un nuevo equipo de desarrollo.
  20. git commit --amend: Pasar la marca --amend a git commit permite modificar la confirmación más reciente. Es muy práctico si olvidas preparar un archivo u omites información importante en el mensaje de confirmación.
